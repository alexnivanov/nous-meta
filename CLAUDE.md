# Nous — географический IRC

Общий контекст для всех под-проектов, вынесенный в отдельный репозиторий. Каждый
под-проект подключает этот файл строкой `@../nous-meta/CLAUDE.md` в своём
`CLAUDE.md` (Claude Code разворачивает `@`-импорт относительно того файла). Здесь
живёт продуктовая идея, модель каналов и **контракт ID каналов** — формат, на
который опираются геокодинг, хранение и клиент. Локальные детали — в `CLAUDE.md`
каждого под-проекта.

## Идея

**Географический IRC / чат как в MMORPG** (WoW, Eve): весь мир разбит на
**вложенные административные каналы**, и пользователь автоматически состоит сразу
в нескольких — от квартала до страны. Чем выше уровень, тем шире аудитория канала.
Главное в проекте — именно **чёткая иерархическая, вложенная структура каналов**.

```
🌍 Страна
 └─ 🗺 Область
     └─ 🏙 Город
         └─ 🏘 Район
             └─ 🏠 Квартал
```

Самый локальный уровень — **Квартал**, но он **опциональный**: в OSM слой
neighbourhood покрыт неравномерно (в центре городов есть, в спальных районах РФ
часто нет). Поэтому **гарантированный локальный пол — Район**, а Квартал
появляется там, где данные есть. Самый локальный канал пользователя = глубочайший
непустой слот (квартал, иначе район). Ниже Квартала не идём: улица — это линия, а
не естественная область (длинная улица проходит через много кварталов), а дом/POI
слишком дробны. Каналы переключаются командами, как в MMORPG:

| Команда  | Уровень |
|----------|---------|
| `/world` | страна  |
| `/city`  | город   |
| `/local` | район   |
| `/say`   | квартал |

Структура **не фиксирована жёстко** — в разных странах админделение различается
(в РФ «область», в ФРГ «Bundesland», города-государства схлопывают уровни). Это не
проблема: набор каналов = реальная цепочка точки, а лишние промежуточные тиры
(федеральный округ в РФ, county в США) отсекаются. Пустой слот = в этой стране
такого уровня просто нет.

## Контракт ID каналов

**Это внутренний контракт сервера и точка опоры клиента:** геокодинг
`nous-server` *производит* ID каналов, его же хранение и роутинг работают *по ним*,
а клиент использует те же ID (переключение каналов, отображение). Все стороны
обязаны одинаково понимать формат — менять его можно только согласованно.

Канал = административная единица. Главное требование: **отображение координат в
каналы однозначно** — одни и те же координаты у двух клиентов всегда дают один и
тот же набор ID, стабильно во времени. Иначе два человека в одном месте попадут в
разные чаты — для чата это фатально.

ID канала — это **не имя, а стабильный код / идентификатор** единицы:

| Уровень | ID канала | Источник |
|---|---|---|
| Страна | ISO 3166-1 (`RU`, `DE`) | `country_code` из OSM |
| Область | ISO 3166-2 (`RU-MOW`, `DE-HE`) | `ISO3166-2-lvl4` из OSM |
| Город / Район / Квартал | `osm_type/osm_id` (`relation/2555133`) | OSM |

Имена («Frankfurt», «Bahnhofsviertel») — **только для отображения**. Совпадения
имён (Москва-город и Москва-область) проблемой не являются: ID у каналов разные,
а в UI различаются подписью уровня.

## Источник данных — OpenStreetMap, не Google

Выбран **OSM/Nominatim**, а не Google. Причины — не только технические:
- **нет завязки на крупную корпорацию** — открытые данные, открытые ID
  (`osm_type/osm_id`), а не проприетарный `place_id`; позиционирование как
  народного, community-owned проекта (OSM сам — волонтёрский);
- **можно поднять свой Nominatim** → независимость от чужого API и реальный оффлайн;
- ISO 3166-2 для области OSM отдаёт сразу (`ISO3166-2-lvl4`), у Google его нет.

## Почему не геохеш (история)

Прежняя идея — ID канала = префикс геохеша. Геохеш давал однозначность и работал
оффлайн, но кодировал **размер ячейки, а не административную границу**: «город»
получался квадратом ~40 км, а не реальным городом, и «один канал на всю страну» на
нём не выразить (страна = тысячи разных префиксов). Раз нужна чёткая админструктура,
а размеры/расстояния не важны, геохеш как идентификатор канала **не используется**.
